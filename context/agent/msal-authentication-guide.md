# MSAL.js Authentication Guide for Amplifier Online

This guide covers implementing MSAL.js authentication for Single-Page Applications (SPAs) deployed
via Amplifier Online. Web frontends always have EasyAuth enforced (RedirectToLoginPage), while
API backends validate tokens via JWT middleware. Frontend applications use MSAL.js to acquire
Bearer tokens for API calls.

---

## Architecture Overview

```
┌────────────────────────────────────────────────────────────┐
│  Browser                                                   │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  SPA (React + MSAL.js)                               │  │
│  │  1. loadAuthConfig() → fetches /auth-config.json     │  │
│  │  2. ssoSilent() → leverages EasyAuth session         │  │
│  │  3. acquireTokenSilent() → gets Bearer token         │  │
│  │  4. fetch(/api/..., {Authorization: Bearer})         │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────┬──────────────────────────────────────────┘
                  │
       ┌──────────┼─────────────────────────────┐
       │          │                             │
  ┌────▼────┐ ┌───▼──────┐         ┌────────────▼──┐
  │ Web ACA │ │ API ACA  │         │ Azure AD      │
  │ (nginx) │ │ (FastAPI)│         │(login.ms)     │
  │ EasyAuth│ │ JWT      │         │               │
  │ (login) │ │ middleware│        │               │
  └─────────┘ └──────────┘         └───────────────┘
```

**Key points:**
- EasyAuth on web frontends handles the initial login and creates an Azure AD session
- MSAL.js must use `ssoSilent()` to leverage the existing session without prompting again
- Frontend acquires Bearer tokens via MSAL.js to call backend APIs
- Backend validates tokens via JWT middleware (signature verification against Entra JWKS endpoint)

## Step 2: Load Runtime Auth Config

**The auth configuration is provided at runtime** (not build time). The Entra app registration
client ID is created during `amplifier-online up` — it doesn't exist when your Docker image is built.

The container's `entrypoint.sh` writes `AUTH_CLIENT_ID` and `AUTH_TENANT_ID` environment variables
to `/auth-config.json` at container startup. Your frontend fetches this when loading.

```typescript
// src/auth/config.ts
import { Configuration } from '@azure/msal-browser';

export async function loadAuthConfig(): Promise<Configuration> {
  // Fetch runtime config (generated by container entrypoint)
  const response = await fetch('/auth-config.json');
  const { clientId, tenantId } = await response.json();

  return {
    auth: {
      clientId,
      authority: `https://login.microsoftonline.com/${tenantId}`,
      redirectUri: window.location.origin + '/',
    },
    cache: {
      cacheLocation: 'localStorage',
      storeAuthStateInCookie: false,
    },
  };
}
```

**Why runtime config?**
- Client ID doesn't exist until deployment
- Same Docker image works across environments (dev, staging, prod)
- No build-time secrets baked into frontend bundle

## Step 3: Initialize MSAL

```typescript
// src/auth/msal.ts
import { PublicClientApplication } from '@azure/msal-browser';
import { loadAuthConfig } from './config';

let msalInstance: PublicClientApplication;

export async function initializeMsal() {
  const config = await loadAuthConfig();
  msalInstance = new PublicClientApplication(config);
  await msalInstance.initialize();
  return msalInstance;
}

export function getMsalInstance() {
  if (!msalInstance) {
    throw new Error('MSAL not initialized. Call initializeMsal() first.');
  }
  return msalInstance;
}
```

**Usage in your app:**
```typescript
// src/App.tsx
useEffect(() => {
  initializeMsal().then(() => {
    console.log('MSAL initialized');
  });
}, []);
```

## Step 4: Handle SSO Silent

**This step is CRITICAL because EasyAuth is always enforced on web frontends** (RedirectToLoginPage).

EasyAuth handles the initial login and creates an Azure AD session. MSAL.js must use `ssoSilent()`
to silently establish an MSAL account from that existing session **without prompting the user to
log in again**.

```typescript
// src/auth/login.ts
import { getMsalInstance } from './msal';

export async function handleLogin() {
  const msalInstance = getMsalInstance();

  try {
    // Try to establish account from existing Azure AD session
    const ssoResult = await msalInstance.ssoSilent({
      scopes: [`api://${msalInstance.getConfiguration().auth.clientId}/access_as_user`],
    });
    
    console.log('SSO Silent succeeded:', ssoResult.account);
    return ssoResult.account;
  } catch (error) {
    // No existing session or SSO failed
    console.log('SSO Silent failed, initiating interactive login:', error);
    
    // Fall back to interactive login
    await msalInstance.loginRedirect({
      scopes: [`api://${msalInstance.getConfiguration().auth.clientId}/access_as_user`],
    });
  }
}
```

**Without `ssoSilent()`:**
- User logs in via EasyAuth redirect
- User gets redirected back to your app
- MSAL.js has no cached account
- User must log in again via MSAL.js (bad UX)

**With `ssoSilent()`:**
- User logs in via EasyAuth redirect (always enforced on web frontends)
- User gets redirected back to your app
- MSAL.js silently establishes account from existing session
- No second login prompt (good UX)

## Step 5: Acquire Access Tokens

Once authenticated, use `acquireTokenSilent()` to get Bearer tokens for API calls:

```typescript
// src/auth/tokens.ts
import { getMsalInstance } from './msal';

export async function getAccessToken(): Promise<string> {
  const msalInstance = getMsalInstance();
  const accounts = msalInstance.getAllAccounts();
  
  if (accounts.length === 0) {
    throw new Error('No accounts found. User must log in.');
  }

  const request = {
    scopes: [`api://${msalInstance.getConfiguration().auth.clientId}/access_as_user`],
    account: accounts[0],
  };

  try {
    // Try silent token acquisition (uses hidden iframe)
    const response = await msalInstance.acquireTokenSilent(request);
    return response.accessToken;
  } catch (error) {
    // Silent acquisition failed, fall back to interactive
    console.log('Silent token acquisition failed, redirecting:', error);
    await msalInstance.acquireTokenRedirect(request);
    throw new Error('Redirecting for interactive token acquisition');
  }
}
```

**How it works:**
- `acquireTokenSilent()` attempts to get a token from cache or via hidden iframe
- If that fails (token expired, cache empty), falls back to interactive redirect
- Returns a valid Bearer token for API calls

## Step 6: Call Your API

Include the Bearer token in the `Authorization` header:

```typescript
// src/api/client.ts
import { getAccessToken } from '../auth/tokens';

export async function callApi(endpoint: string, options: RequestInit = {}) {
  const accessToken = await getAccessToken();

  const response = await fetch(`/api${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      'Authorization': `Bearer ${accessToken}`,
      'Content-Type': 'application/json',
    },
  });

  if (!response.ok) {
    throw new Error(`API call failed: ${response.statusText}`);
  }

  return response.json();
}

// Usage
const data = await callApi('/users/me');
```

**Backend validation:**
Your backend validates the Bearer token via JWT middleware (`jwt_middleware.py`). The middleware
verifies the token signature against Entra's JWKS endpoint, checks audience and expiry, and
extracts user identity (oid, name, roles, groups) from JWT claims. See the "Backend: Reading
the Authenticated User" section below.

---

## Backend: Reading the Authenticated User

The JWT middleware validates the Bearer token and makes user claims available on the request.
API services use `jwt_middleware.py` (not EasyAuth) for token validation.

```python
# FastAPI example using JWT middleware
# The middleware validates the token and attaches claims to request.state.user
from fastapi import Request, HTTPException

@app.get("/api/me")
def get_current_user(request: Request):
    user = getattr(request.state, "user", None)
    if not user:
        raise HTTPException(status_code=401, detail="Not authenticated")
    
    return {
        "user": {
            "user_id": user.oid,                    # Object ID (user's unique ID)
            "email": user.preferred_username,        # User's email/UPN
            "name": user.name,                       # Display name
            "groups": user.groups,                   # Security group IDs
            "roles": user.roles,                     # App role assignments
        }
    }
```

**Setup:** Add the JWT middleware to your FastAPI app:
```python
from middleware.jwt_middleware import JWTAuthMiddleware
app.add_middleware(JWTAuthMiddleware)
```

The middleware reads `AUTH_CLIENT_ID` and `AUTH_TENANT_ID` from environment variables
(auto-injected by the platform) and validates tokens against Entra's JWKS endpoint.

## Group-Based Authorization

**Security group memberships are included in the JWT token** as group object IDs.
The JWT middleware extracts these from the token claims automatically.

```python
# Python example using JWT middleware claims
from fastapi import Request, HTTPException
import os

# Get admin group ID from environment (set by platform admin)
ADMIN_GROUP_ID = os.getenv("ADMIN_GROUP_ID")  # Ask your platform admin for this value

def require_group(required_group_id: str):
    """Dependency to require group membership via JWT claims."""
    def checker(request: Request):
        user = getattr(request.state, "user", None)
        if not user:
            raise HTTPException(status_code=401, detail="Not authenticated")
        
        if required_group_id not in user.groups:
            raise HTTPException(status_code=403, detail="Insufficient permissions")
        
        return user
    return checker

# Use in endpoint
@app.post("/api/admin/delete-user")
async def delete_user(user_id: str, request: Request):
    require_group(ADMIN_GROUP_ID)(request)
    # Only members of ADMIN_GROUP_ID can call this
    return {"deleted": user_id}
```

**Important:** Group membership is evaluated at **token issuance time**. If a user's group
membership changes:
1. Their existing tokens still have the old groups
2. They must acquire a new token to see the updated groups
3. See "Token Cache Clearing" section below

## Token Cache Clearing After Group Changes

**Problem:** You changed a user's group membership (added/removed from a security group), but
the app still shows old permissions.

**Why:** MSAL.js caches tokens in localStorage. The cached access token still contains the **old**
group claims. New group memberships only appear in **newly issued** tokens.

**Solution:** Clear the MSAL token cache and force a new login:

```typescript
// src/auth/logout.ts
import { getMsalInstance } from './msal';

export async function clearTokenCache() {
  const msalInstance = getMsalInstance();
  
  // Clear all accounts and tokens
  const accounts = msalInstance.getAllAccounts();
  for (const account of accounts) {
    await msalInstance.logout({ account });
  }
  
  // Clear localStorage entirely (MSAL cache + any app state)
  localStorage.clear();
  
  // Redirect to login
  window.location.href = '/';
}
```

**When to use:**
- After changing user group memberships
- After changing token claims configuration
- When troubleshooting stale permissions

**Alternative:** Provide a "Refresh Permissions" button in your UI that calls `clearTokenCache()`.

## Common Issues and Solutions

### Issue: Authentication Redirect Loops

**Symptom:** App redirects to login.microsoftonline.com and back repeatedly.

**Cause:** Redirect URIs in Entra app registration don't match your Container App or Static Web
App URLs.

**Solution:**
1. Verify redirect URIs in Azure Portal → App Registrations → your app → Redirect URIs
2. Re-run `amplifier-online up` to reconfigure redirect URIs automatically
3. Ensure `redirectUri` in MSAL config matches your app's actual URL

### Issue: "User is blocked from accessing the application" (AADSTS50105)

**Symptom:** User gets blocked error during login.

**Cause:** User is not a direct member of the security group assigned to the Entra app.

**Solution:**
1. Verify the user is a **direct member** of the "Amplifier Online Users" group (or equivalent)
2. **Nested group membership does NOT work** — Entra app role assignments don't traverse group hierarchies
3. Ask your platform administrator to add the user as a direct member

### Issue: Token Contains Old Group Memberships

**Symptom:** User's permissions don't reflect recent group changes.

**Cause:** Token was issued before group membership change. MSAL cache still has the old token.

**Solution:**
1. Call `clearTokenCache()` (see "Token Cache Clearing" section above)
2. User logs in again and gets a new token with updated groups

### Issue: CORS Errors When Calling API

**Symptom:** Frontend can't call backend due to CORS errors.

**Cause:** Backend doesn't allow frontend origin in CORS configuration.

**Solution (for web-app-awa and external APIs):**
```python
# FastAPI example
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "https://*.azurestaticapps.net",  # Production
        "http://localhost:3000",          # Local development
    ],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**Note:** For `web-app-aca` stacks, frontend and backend are on the same domain (internal
routing), so CORS is usually not needed.

## Environment Variables

Your containers automatically receive these authentication environment variables:

| Variable | Description | Example |
|----------|-------------|---------|
| `AUTH_CLIENT_ID` | Your project's Entra app registration client ID | `a1b2c3d4-...` |
| `AUTH_TENANT_ID` | Microsoft tenant ID | `72f988bf-...` |

These are written to `/auth-config.json` by the container entrypoint for frontend consumption.

**Static Web App frontends (`web-app-awa`, `static-web-app`):** Frontends always have
`protected: login` enforced, which implies `auth: true`. The orchestrator registers an Entra
app and injects `AZURE_CLIENT_ID` into the Static Web App environment. Your SPA reads the
client ID from the SWA environment rather than `/auth-config.json`.

**API services:** Use `AUTH_CLIENT_ID` to configure the JWT middleware audience
(`api://{AUTH_CLIENT_ID}`) and `AUTH_TENANT_ID` to derive the JWKS endpoint URL.

> **Note:** This `AZURE_CLIENT_ID` is the **per-project Entra app registration** used by
> MSAL.js for end-user authentication. It is unrelated to CI/CD. Container CI/CD workflows
> do not use `AZURE_CLIENT_ID` — they authenticate via GitHub Actions OIDC tokens validated
> directly by the provisioner. See the [CI/CD Guide](cicd-guide.md) for details.

## Complete Example: React App with MSAL.js

```typescript
// src/App.tsx
import React, { useEffect, useState } from 'react';
import { initializeMsal } from './auth/msal';
import { handleLogin } from './auth/login';
import { callApi } from './api/client';

function App() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    initializeMsal()
      .then(() => handleLogin())
      .then((account) => {
        setUser(account);
        setLoading(false);
      })
      .catch((error) => {
        console.error('Auth error:', error);
        setLoading(false);
      });
  }, []);

  const fetchUserData = async () => {
    try {
      const data = await callApi('/users/me');
      console.log('User data:', data);
    } catch (error) {
      console.error('API call failed:', error);
    }
  };

  if (loading) {
    return <div>Loading...</div>;
  }

  return (
    <div>
      <h1>Welcome, {user?.name}</h1>
      <button onClick={fetchUserData}>Fetch My Data</button>
    </div>
  );
}

export default App;
```
