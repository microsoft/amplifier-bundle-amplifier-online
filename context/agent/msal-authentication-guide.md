# MSAL.js Authentication Guide for Amplifier Online

This guide covers implementing MSAL.js authentication for Single-Page Applications (SPAs) deployed
via Amplifier Online. Web frontends always have EasyAuth enforced (RedirectToLoginPage), while
API backends validate tokens via JWT middleware. Frontend applications use MSAL.js to acquire
Bearer tokens for API calls.

**Two registrations, split by OAuth role.** A project no longer has one `ao-{project}` app. The
platform creates registrations per role: `ao-{project}-client` (the LOGIN client — public PKCE,
holds redirect URIs, the app the SPA/EasyAuth sign in AS; created when the project has a frontend)
and `ao-{project}-api` (the API AUDIENCE the backend validates — `api://{apiClientId}`; created when
the project hosts a non-`web` backend service). The `-api` exposes the `access_as_user` scope and is
APIM-fronted only when user-facing (`has a frontend` or `auth.expose: true`); an internal-only API
exposes no scope and no APIM. The old suffix-less `ao-{project}` / `ao-{project}-app` names are gone.

---

## Architecture Overview

```
┌────────────────────────────────────────────────────────────┐
│  Browser                                                   │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  SPA (React + MSAL.js)                               │  │
│  │  1. loadAuthConfig() → fetches /auth-config.json     │  │
│  │  2. ssoSilent() → leverages EasyAuth session         │  │
│  │  3. acquireTokenSilent() → gets Bearer token         │  │
│  │  4. fetch(/api/..., {Authorization: Bearer})         │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────┬──────────────────────────────────────────┘
                  │
       ┌──────────┼─────────────────────────────┐
       │          │                             │
  ┌────▼────┐ ┌───▼──────┐         ┌────────────▼──┐
  │ Web ACA │ │ API ACA  │         │ Azure AD      │
  │ (nginx) │ │ (FastAPI)│         │(login.ms)     │
  │ EasyAuth│ │ JWT      │         │               │
  │ (login) │ │ middleware│        │               │
  └─────────┘ └──────────┘         └───────────────┘
```

**Key points:**
- EasyAuth on web frontends handles the initial login and creates an Azure AD session
- MSAL.js must use `ssoSilent()` to leverage the existing session without prompting again
- Frontend acquires Bearer tokens via MSAL.js to call backend APIs
- Backend validates tokens via JWT middleware (signature verification against Entra JWKS endpoint)

## Step 2: Load Runtime Auth Config

**The auth configuration is provided at runtime** (not build time). The Entra app registrations are
created during `amplifier-online up` — they don't exist when your Docker image is built.

A frontend receives **two ids**: `AZURE_CLIENT_ID` (the `-client` appId — who you sign in AS) and
`AZURE_API_CLIENT_ID` (the `-api` appId — the token AUDIENCE). The container's `entrypoint.sh` writes
these plus `AZURE_TENANT_ID` (and any `consumes` map) to `/auth-config.json` (keys `clientId`,
`apiClientId`, `tenantId`, `consumes`) at startup. Your frontend fetches this when loading.

The MSAL config signs in AS `clientId`, but the API scope is requested against `apiClientId` —
`api://{apiClientId}/access_as_user`, **not** `api://{clientId}/...`. The login client and the token
audience are now different registrations.

```typescript
// src/auth/config.ts
import { Configuration } from '@azure/msal-browser';

export interface AuthConfig extends Configuration {
  apiClientId?: string;
  consumes?: Record<string, { apiClientId: string; baseUrl: string }>;
}

export async function loadAuthConfig(): Promise<AuthConfig> {
  // Fetch runtime config (generated by container entrypoint)
  const response = await fetch('/auth-config.json');
  const { clientId, apiClientId, tenantId, consumes } = await response.json();

  return {
    auth: {
      clientId,                          // the -client appId (who you sign in AS)
      authority: `https://login.microsoftonline.com/${tenantId}`,
      redirectUri: window.location.origin + '/',
    },
    apiClientId,                         // the -api appId (the audience for access_as_user)
    consumes,
    cache: {
      cacheLocation: 'localStorage',
      storeAuthStateInCookie: false,
    },
  };
}

// Scope helper: prefer the API audience, fall back to the client itself.
export function getApiScope(config: AuthConfig): string {
  return `api://${config.apiClientId ?? config.clientId}/access_as_user`;
}
```

**Why runtime config?**
- App ids don't exist until deployment
- Same Docker image works across environments (dev, staging, prod)
- No build-time secrets baked into frontend bundle

> **Bring Your Own App Registration (per role):** If your team already has Entra registrations, skip
> provisioner-managed creation by adding either/both `auth.api_app_id` and `auth.client_app_id` to
> `amplifier-online.yaml` (the old single `auth.client_id` field is removed). Validation is read-only
> (never modifies) and role-aware: a `client_app_id` is **not** required to expose `access_as_user`;
> an `api_app_id` needs identifier URIs + an `access_as_user` scope only when user-facing. BYO
> registrations are skipped on destroy.
>
> ```yaml
> auth:
>   api_app_id: "<existing -api appId>"        # optional
>   client_app_id: "<existing -client appId>"  # optional
> ```

## Step 3: Initialize MSAL

```typescript
// src/auth/msal.ts
import { PublicClientApplication } from '@azure/msal-browser';
import { loadAuthConfig, AuthConfig } from './config';

let msalInstance: PublicClientApplication;

// Pass a config (loaded once via loadAuthConfig) or let it load its own.
export async function initializeMsal(config?: AuthConfig) {
  const cfg = config ?? await loadAuthConfig();
  msalInstance = new PublicClientApplication(cfg);
  await msalInstance.initialize();
  return msalInstance;
}

export function getMsalInstance() {
  if (!msalInstance) {
    throw new Error('MSAL not initialized. Call initializeMsal() first.');
  }
  return msalInstance;
}
```

**Usage in your app:**
```typescript
// src/App.tsx
useEffect(() => {
  initializeMsal().then(() => {
    console.log('MSAL initialized');
  });
}, []);
```

## Step 4: Handle SSO Silent

**This step is CRITICAL because EasyAuth is always enforced on web frontends** (RedirectToLoginPage).

EasyAuth handles the initial login and creates an Azure AD session. MSAL.js must use `ssoSilent()`
to silently establish an MSAL account from that existing session **without prompting the user to
log in again**.

```typescript
// src/auth/login.ts
import { getMsalInstance } from './msal';

export async function handleLogin(apiScope: string) {
  const msalInstance = getMsalInstance();

  try {
    // Try to establish account from existing Azure AD session.
    // apiScope is getApiScope(config) → api://{apiClientId}/access_as_user
    const ssoResult = await msalInstance.ssoSilent({ scopes: [apiScope] });
    
    console.log('SSO Silent succeeded:', ssoResult.account);
    return ssoResult.account;
  } catch (error) {
    // No existing session or SSO failed
    console.log('SSO Silent failed, initiating interactive login:', error);
    
    // Fall back to interactive login
    await msalInstance.loginRedirect({ scopes: [apiScope] });
  }
}
```

**Without `ssoSilent()`:**
- User logs in via EasyAuth redirect
- User gets redirected back to your app
- MSAL.js has no cached account
- User must log in again via MSAL.js (bad UX)

**With `ssoSilent()`:**
- User logs in via EasyAuth redirect (always enforced on web frontends)
- User gets redirected back to your app
- MSAL.js silently establishes account from existing session
- No second login prompt (good UX)

## Step 5: Acquire Access Tokens

Once authenticated, use `acquireTokenSilent()` to get Bearer tokens for API calls:

```typescript
// src/auth/tokens.ts
import { getMsalInstance } from './msal';

export async function getAccessToken(apiScope: string): Promise<string> {
  const msalInstance = getMsalInstance();
  const accounts = msalInstance.getAllAccounts();
  
  if (accounts.length === 0) {
    throw new Error('No accounts found. User must log in.');
  }

  const request = {
    // apiScope is getApiScope(config) → api://{apiClientId}/access_as_user
    scopes: [apiScope],
    account: accounts[0],
  };

  try {
    // Try silent token acquisition (uses hidden iframe)
    const response = await msalInstance.acquireTokenSilent(request);
    return response.accessToken;
  } catch (error) {
    // Silent acquisition failed, fall back to interactive
    console.log('Silent token acquisition failed, redirecting:', error);
    await msalInstance.acquireTokenRedirect(request);
    throw new Error('Redirecting for interactive token acquisition');
  }
}
```

**How it works:**
- `acquireTokenSilent()` attempts to get a token from cache or via hidden iframe
- If that fails (token expired, cache empty), falls back to interactive redirect
- Returns a valid Bearer token for API calls

## Step 6: Call Your API

Include the Bearer token in the `Authorization` header:

```typescript
// src/api/client.ts
import { getAccessToken } from '../auth/tokens';

export async function callApi(endpoint: string, apiScope: string, options: RequestInit = {}) {
  const accessToken = await getAccessToken(apiScope);

  const response = await fetch(`/api${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      'Authorization': `Bearer ${accessToken}`,
      'Content-Type': 'application/json',
    },
  });

  if (!response.ok) {
    throw new Error(`API call failed: ${response.statusText}`);
  }

  return response.json();
}

// Usage (apiScope = getApiScope(config) → api://{apiClientId}/access_as_user)
const data = await callApi('/users/me', apiScope);
```

**Backend validation:**
Your backend validates the Bearer token via JWT middleware (`jwt_middleware.py`). The middleware
verifies the token signature against Entra's JWKS endpoint, checks audience and expiry, and
extracts user identity (oid, name, roles, groups) from JWT claims. See the "Backend: Reading
the Authenticated User" section below.

## Calling Another Project's API (cross-project consumption)

To call a different project's API, the producer sets `auth.expose: true` (its `-api` becomes
user-facing — gains `access_as_user` + APIM — even with no frontend); the consumer lists it under
`auth.consumes`. The platform wires the consumer's `-client` requiredResourceAccess to the producer's
`access_as_user` and pre-authorizes the consumer on the producer (**producer must be deployed first**).

```yaml
# producer's amplifier-online.yaml
auth: { expose: true }

# consumer's amplifier-online.yaml
auth:
  consumes:
    - other-project                                  # by deployed project name, or:
    - { api_app_id: "<appId>", base_url: "https://..." }
```

The producer map reaches the SPA via `auth-config.json`'s `consumes` (ACA) or `VITE_AO_CONSUMES`
(SWA). Helper `getConsumedApi(config, name)` returns `{ scope, baseUrl }`; request that `scope` from
MSAL and fetch `baseUrl`.

## Backend: Reading the Authenticated User

The JWT middleware validates the Bearer token and makes user claims available on the request.
API services use `jwt_middleware.py` (not EasyAuth) for token validation.

```python
# FastAPI example using JWT middleware
# The middleware validates the token and attaches claims to request.state.user
from fastapi import Request, HTTPException

@app.get("/api/me")
def get_current_user(request: Request):
    user = getattr(request.state, "user", None)
    if not user:
        raise HTTPException(status_code=401, detail="Not authenticated")
    
    return {
        "user": {
            "user_id": user.oid,                    # Object ID (user's unique ID)
            "email": user.preferred_username,        # User's email/UPN
            "name": user.name,                       # Display name
            "groups": user.groups,                   # Security group IDs
            "roles": user.roles,                     # App role assignments
        }
    }
```

**Setup:** Add the JWT middleware to your FastAPI app:
```python
from middleware.jwt_middleware import JWTAuthMiddleware
app.add_middleware(JWTAuthMiddleware)
```

The middleware reads `AZURE_CLIENT_ID` and `AZURE_TENANT_ID` from environment variables
(auto-injected by the platform) and validates tokens against Entra's JWKS endpoint. On a backend,
`AZURE_CLIENT_ID` is the **`-api` appId**, and the middleware enforces `aud == api://{AZURE_CLIENT_ID}`.

## Group-Based Authorization

**Security group memberships are included in the JWT token** as group object IDs.
The JWT middleware extracts these from the token claims automatically.

```python
# Python example using JWT middleware claims
from fastapi import Request, HTTPException
import os

# Get admin group ID from environment (set by platform admin)
ADMIN_GROUP_ID = os.getenv("ADMIN_GROUP_ID")  # Ask your platform admin for this value

def require_group(required_group_id: str):
    """Dependency to require group membership via JWT claims."""
    def checker(request: Request):
        user = getattr(request.state, "user", None)
        if not user:
            raise HTTPException(status_code=401, detail="Not authenticated")
        
        if required_group_id not in user.groups:
            raise HTTPException(status_code=403, detail="Insufficient permissions")
        
        return user
    return checker

# Use in endpoint
@app.post("/api/admin/delete-user")
async def delete_user(user_id: str, request: Request):
    require_group(ADMIN_GROUP_ID)(request)
    # Only members of ADMIN_GROUP_ID can call this
    return {"deleted": user_id}
```

**Important:** Group membership is evaluated at **token issuance time**. If a user's group
membership changes:
1. Their existing tokens still have the old groups
2. They must acquire a new token to see the updated groups
3. See "Token Cache Clearing" section below

## Token Cache Clearing After Group Changes

**Problem:** You changed a user's group membership (added/removed from a security group), but
the app still shows old permissions.

**Why:** MSAL.js caches tokens in localStorage. The cached access token still contains the **old**
group claims. New group memberships only appear in **newly issued** tokens.

**Solution:** Clear the MSAL token cache and force a new login:

```typescript
// src/auth/logout.ts
import { getMsalInstance } from './msal';

export async function clearTokenCache() {
  const msalInstance = getMsalInstance();
  
  // Clear all accounts and tokens
  const accounts = msalInstance.getAllAccounts();
  for (const account of accounts) {
    await msalInstance.logout({ account });
  }
  
  // Clear localStorage entirely (MSAL cache + any app state)
  localStorage.clear();
  
  // Redirect to login
  window.location.href = '/';
}
```

**When to use:**
- After changing user group memberships
- After changing token claims configuration
- When troubleshooting stale permissions

**Alternative:** Provide a "Refresh Permissions" button in your UI that calls `clearTokenCache()`.

## Common Issues and Solutions

### Issue: Authentication Redirect Loops

**Symptom:** App redirects to login.microsoftonline.com and back repeatedly.

**Cause:** Redirect URIs in Entra app registration don't match your Container App or Static Web
App URLs.

**Solution:**
1. Verify redirect URIs in Azure Portal → App Registrations → your app → Redirect URIs
2. Re-run `amplifier-online up` to reconfigure redirect URIs automatically
3. Ensure `redirectUri` in MSAL config matches your app's actual URL

### Issue: "User is blocked from accessing the application" (AADSTS50105)

**Symptom:** User gets blocked error during login.

**Cause:** User is not a direct member of the security group assigned to the Entra app.

**Solution:**
1. Verify the user is a **direct member** of the "Amplifier Online Users" group (or equivalent)
2. **Nested group membership does NOT work** — Entra app role assignments don't traverse group hierarchies
3. Ask your platform administrator to add the user as a direct member

### Issue: Token Contains Old Group Memberships

**Symptom:** User's permissions don't reflect recent group changes.

**Cause:** Token was issued before group membership change. MSAL cache still has the old token.

**Solution:**
1. Call `clearTokenCache()` (see "Token Cache Clearing" section above)
2. User logs in again and gets a new token with updated groups

### Issue: CORS Errors When Calling API

**Symptom:** Frontend can't call backend due to CORS errors.

**Cause:** Backend doesn't allow frontend origin in CORS configuration.

**Solution (for web-app-awa and external APIs):**
```python
# FastAPI example
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "https://*.azurestaticapps.net",  # Production
        "http://localhost:3000",          # Local development
    ],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**Note:** For `web-app-aca` stacks, frontend and backend are on the same domain (internal
routing), so CORS is usually not needed.

## Environment Variables

Your containers automatically receive these authentication environment variables:

| Variable | Where | Description | Example |
|----------|-------|-------------|---------|
| `AZURE_CLIENT_ID` | frontend | the `-client` appId — who the SPA signs in AS | `a1b2c3d4-...` |
| `AZURE_API_CLIENT_ID` | frontend | the `-api` appId — audience for `access_as_user` | `e5f6...` |
| `AZURE_CLIENT_ID` | backend | the `-api` appId — validated audience (`api://{id}`) | `e5f6...` |
| `AZURE_TENANT_ID` | both | Microsoft tenant ID | `72f988bf-...` |

The frontend ids are written to `/auth-config.json` by the container entrypoint (keys `clientId`,
`apiClientId`, `tenantId`, `consumes`).

**Static Web App frontends (`web-app-awa`, `static-web-app`):** Frontends always have
`protected: login` enforced. SWAs are **client-only** — they get a
`-client` registration but no `-api` and no `access_as_user`; they call external APIs (e.g.
Microsoft Graph) with those APIs' own scopes. `amplifier-online cicd create` injects
`VITE_AZURE_CLIENT_ID` and `VITE_AZURE_TENANT_ID` (plus `VITE_AZURE_API_CLIENT_ID` / `VITE_AO_CONSUMES`
when applicable) as GitHub Actions **build-time** variables. Your SPA reads these via
`import.meta.env.VITE_AZURE_CLIENT_ID` (baked into the JS bundle at build time) rather than the
`/auth-config.json` runtime fetch pattern used by container stacks. See "SWA Frontend Auth
Configuration" section below for details.

**API services:** Use `AZURE_CLIENT_ID` (the `-api` appId) to configure the JWT middleware audience
(`api://{AZURE_CLIENT_ID}`) and `AZURE_TENANT_ID` to derive the JWKS endpoint URL.

> **Note:** The frontend's `AZURE_CLIENT_ID` is the **per-project `-client` Entra registration** used
> by MSAL.js for end-user authentication. Do **not** confuse it with the CI/CD DEPLOY identity's
> `AZURE_CLIENT_ID`, which is a separate GitHub-OIDC app. Container CI/CD workflows authenticate via
> GitHub Actions OIDC tokens validated directly by the provisioner. See the
> [CI/CD Guide](cicd-guide.md) for details.

## SWA Frontend Auth Configuration (Build-Time Injection)

**This section applies to `static-web-app` and `web-app-awa` frontend stacks only.**

Unlike container stacks (which use the runtime `/auth-config.json` fetch pattern described in
Step 2 above), SWA-hosted SPAs receive auth configuration at **build time** as GitHub Actions
variables. SWAs are **client-only** (a `-client` registration, no `-api`, no `access_as_user`).

### How it works

1. `amplifier-online up` creates the `-client` Entra registration and provisions the SWA resource
2. `amplifier-online cicd create` sets `VITE_AZURE_CLIENT_ID` and `VITE_AZURE_TENANT_ID` (plus
   `VITE_AO_CONSUMES` when the SWA consumes other-project APIs) as GitHub Actions build-time variables
3. During the GitHub Actions build, these are exposed as env vars
4. Vite (or similar build tools) bakes `VITE_*` env vars into the JavaScript bundle
5. At runtime, the values are already embedded in the JS -- no fetch needed

### SWA auth config pattern

```typescript
// src/auth/config.ts (for SWA-hosted SPAs)
import { Configuration } from '@azure/msal-browser';

// Values are baked in at build time by Vite from SWA app settings.
// No runtime fetch needed -- unlike the /auth-config.json pattern for containers.
export const msalConfig: Configuration = {
  auth: {
    clientId: import.meta.env.VITE_AZURE_CLIENT_ID ?? '',
    authority: `https://login.microsoftonline.com/${import.meta.env.VITE_AZURE_TENANT_ID}`,
    redirectUri: window.location.origin + '/',
  },
  cache: {
    cacheLocation: 'localStorage',
    storeAuthStateInCookie: false,
  },
};
```

### Container vs SWA auth config comparison

| | Container stacks (`web-app-aca`) | SWA stacks (`static-web-app`, `web-app-awa`) |
|--|--|--|
| **When injected** | Runtime (container start) | Build time (GitHub Actions variables) |
| **Env var names** | `AZURE_CLIENT_ID`, `AZURE_API_CLIENT_ID`, `AZURE_TENANT_ID` | `VITE_AZURE_CLIENT_ID`, `VITE_AZURE_TENANT_ID` (client-only; no `-api`) |
| **How SPA reads** | Fetch `/auth-config.json` at page load | `import.meta.env.VITE_AZURE_CLIENT_ID` (baked into JS) |
| **Async?** | Yes -- `loadAuthConfig()` returns a Promise | No -- values are static imports |

> **Common mistake:** Using `VITE_MSAL_CLIENT_ID` or other custom env var names. The platform
> injects `VITE_AZURE_CLIENT_ID` and `VITE_AZURE_TENANT_ID` -- your code must use these exact
> names or the values will be empty strings at runtime.

## SWA Authentication via `staticwebapp.config.json`

Static Web App (SWA) stacks (`web-app-awa`, `static-web-app`) use `staticwebapp.config.json`
for authentication instead of the EasyAuth sidecar used by ACA stacks.

**How it works:**
- `amplifier-online init` generates `staticwebapp.config.json` automatically for SWA stacks
- The file configures the AAD identity provider with `openIdIssuer` pinned to the tenant
  and `clientIdSettingName: "AZURE_CLIENT_ID"`
- Key settings are `AZURE_CLIENT_ID` (the SWA's `-client` appId) and `AZURE_TENANT_ID`

**Troubleshooting:** If authentication isn't working on a SWA deployment, verify that
`staticwebapp.config.json` contains an `auth` section with the AAD identity provider
configured. Missing or incomplete `auth` configuration is the most common cause of SWA
auth failures.

## Complete Example: React App with MSAL.js

```typescript
// src/App.tsx
import React, { useEffect, useState } from 'react';
import { loadAuthConfig, getApiScope } from './auth/config';
import { initializeMsal } from './auth/msal';
import { handleLogin } from './auth/login';
import { callApi } from './api/client';

function App() {
  const [user, setUser] = useState(null);
  const [apiScope, setApiScope] = useState('');
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    loadAuthConfig()
      .then(async (config) => {
        const scope = getApiScope(config);   // api://{apiClientId}/access_as_user
        setApiScope(scope);
        await initializeMsal(config);
        return handleLogin(scope);
      })
      .then((account) => {
        setUser(account);
        setLoading(false);
      })
      .catch((error) => {
        console.error('Auth error:', error);
        setLoading(false);
      });
  }, []);

  const fetchUserData = async () => {
    try {
      const data = await callApi('/users/me', apiScope);
      console.log('User data:', data);
    } catch (error) {
      console.error('API call failed:', error);
    }
  };

  if (loading) {
    return <div>Loading...</div>;
  }

  return (
    <div>
      <h1>Welcome, {user?.name}</h1>
      <button onClick={fetchUserData}>Fetch My Data</button>
    </div>
  );
}

export default App;
```
